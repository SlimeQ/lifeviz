# Rendering Pipeline

## Layout

- `MainWindow.xaml` hosts a `Viewbox`-wrapped render host so the simulation keeps a constant aspect ratio. Default is 16:9, but the primary source in the *Sources* stack overrides the ratio automatically (if the primary is a layer group, its first child supplies the ratio).
- The logical output surface is still sized to the current simulation grid (`columns x rows`), so each logical cell maps to a single pixel before scaling; that surface is presented either through a D3D11 `DrawingSurface` (preferred) or the legacy `WriteableBitmap` fallback.
- Presentation is now routed through a dedicated WPF CPU presentation backend (`WpfPresentationBackend`) that owns surface allocation, passthrough underlay presentation, shader-fallback effect state, and final pixel upload.
- CPU source compositing is also isolated into its own component (`CpuSourceCompositor`) instead of being embedded directly inside `MainWindow`.
- `MainWindow` now addresses rendering through an explicit `IRenderBackend` contract. The default startup path attempts `GpuRenderBackend` first, which swaps the XAML image host to a `Vortice.Wpf.DrawingSurface`, uploads simulation/underlay buffers into D3D11 textures, resolves the final passthrough composite in a fullscreen GPU shader pass, and now also prefers a D3D11 source-composite pass for `BuildCompositeFrame`; `CpuRenderBackend` remains as migration fallback.
- Final simulation-layer compositing now also prefers a dedicated D3D11 fullscreen pass in `GpuPresentationBackend` when not recording. Active simulation layers are presented with per-layer blend mode/opacity metadata, and the old CPU per-pixel blend in `MainWindow.RenderFrame()` remains as the fallback path for unsupported cases and recording.
- The render host (`Viewbox` / `RenderSurfaceHost` / `DrawingSurface` fallback image) is explicitly non-hit-testable. Mouse input should bypass the presentation surface and land on the window/root context-menu layer instead of forcing WPF to interact with the GPU host on hover.
- Simulation layers now target an `ISimulationBackend` contract, but the runtime implementation is now GPU-only: `GpuSimulationBackend` is required at startup and the old `CpuSimulationBackend` fallback has been removed.
- `GpuSourceCompositor` and `GpuSimulationBackend` now share a single D3D11 device/context (`GpuSharedDevice`) so the source composite can be handed straight to either grayscale or RGB simulation layers as a shader resource instead of forcing a CPU mask rebuild first.
- Native-source preference is now differentiated by media type. Static file sources can return to native frames at moderate output sizes, but video file sources and video sequences stay on adapted/direct decoder output across the preset ladder. The earlier video-native fallback was the point where `1080p+` underlay freshness collapsed even when render/sim timing still looked acceptable.
- `GpuSourceCompositor` now explicitly unbinds its composite render target before returning that texture to the simulation backend. This avoids a D3D11 SRV/RTV hazard that could otherwise make RGB composite-threshold injection sample black even when the source composite itself was correct.
- Smoke-test startup now runs under `App.IsSmokeTestMode`, which still constructs the real WPF window/render backends but skips persisted project load, media/file/window source startup, ffmpeg decode, and audio-loopback initialization so startup regressions can be tested without booting the full user scene.
- Main window shutdown now stops the background frame timer and swaps to a null render backend before disposing GPU/CPU render resources, so teardown does not race a final UI-frame callback against already-freed backend state.
- The GPU presentation backend also exposes smoke-test counters so `--smoke-test gpu-render` can assert that the fullscreen composite shader path actually initialized through `MainWindow`, not just that a hidden window opened successfully.
- The GPU source compositor also exposes smoke-test counters so `--smoke-test gpu-source` can assert that `BuildCompositeFrame` actually ran through the D3D11 path instead of silently falling back to CPU.
- `--smoke-test gpu-handoff` verifies that the GPU source composite can inject directly into the GPU sim path through `MainWindow` and that the composite frame carries zero CPU readback bytes for that test.
- `--smoke-test gpu-rgb-threshold` specifically validates the RGB composite-threshold injection path with a pure-white source and fails unless all three RGB channels inject successfully.
- `--smoke-test passthrough-underlay-only` now drives the real frame tick for the zero-layer passthrough case: when all simulation layers are disabled, source capture/composite refresh must still happen and the final frame must still present the source composite instead of dropping to black.
- `--smoke-test gpu-frequency-hue` validates the per-layer frequency-driven hue path numerically: it checks that changing dominant frequency changes the hue metadata sent to final composition while the raw RGB simulation buffer remains unchanged on the live GPU path.
- `--smoke-test simulation-reactive-mappings` validates the modular per-layer mapping system directly: it checks `Level -> Opacity`, `Level -> Framerate`, `Frequency -> Hue Shift`, `Injection Noise`, `Threshold Min`, `Threshold Max`, and `Hue Speed` against known normalized audio inputs, and it does so with the legacy global audio-reactivity master toggle forced off so per-layer mappings cannot silently depend on the old global gate.
- `--smoke-test simulation-reactive-persistence` round-trips per-layer reactive mappings through both the scene-project serializer (`LayerConfigFile`) and the app-config normalization path, so save/load regressions in the new mapping system are caught automatically.
- `--smoke-test simulation-reactive-legacy-migration` verifies that old whole-scene `Level -> Framerate` / `Level -> Life Opacity` settings are converted into per-layer mappings and that the legacy flags are cleared afterward.
- `--smoke-test gpu-injection-mode` validates that the GPU composite-injection path still distinguishes `Threshold`, `Random Pulse`, and `Pulse Width Modulation` on a mid-gray source instead of collapsing them to the same result.
- `--smoke-test gpu-file-injection-mode <video>` runs that same injection-mode density check against the first decoded frame of a real file source, which is useful when synthetic sources do not reproduce a report.
- The combined `--smoke-test all` GPU UI suite now also verifies that the passthrough-underlay + sim-layer live path still renders correctly after GPU presentation changes.
- `--smoke-test gpu-benchmark` uses the same counters plus direct timing around `GpuSimulationBackend` to separate GPU source-composite upload/draw/readback cost from GPU sim inject/step/fill cost, which is useful for deciding whether the next optimization should target readback elimination or more sim work on GPU.
- `--smoke-test profile-mainloop` runs the real background-timer frame loop on a hidden synthetic scene and captures stage timings for capture/composite/inject/step/render/present plus dispatcher input/background latency samples. It also records frame-gap spike metrics (`frame_gap_over_25ms`, `frame_gap_over_33ms`, `frame_gap_over_50ms`) so pacing regressions can be separated from average throughput problems. Profile/pacing smokes now disable smoke-only CPU fallback color-buffer capture for GPU simulation layers, so render timings reflect the live shared-texture path instead of the correctness-validation path. Normal app runs export JSON reports under `%LOCALAPPDATA%\lifeviz\profiles`; smoke-test runs export under `bin\Debug\net9.0-windows-sbx\profiles\`.
- `--smoke-test profile-240` / `profile-480` pin that profiler to grayscale scenes at fixed resolutions, while `profile-rgb-240` / `profile-rgb-480` force the reference simulation layer into `RGB Channel Bins` so RGB-path regressions can be measured directly.
- `--smoke-test profile-file-240 <video>` / `profile-file-480 <video>` swap the synthetic source stack for a real file-backed source so decode/playback jitter can be profiled with the same stage timing pipeline. `profile-file-rgb-240 <video>` / `profile-file-rgb-480 <video>` do the same with the reference simulation layer pinned to `RGB Channel Bins`, which is the closest built-in smoke to the user's "real media + RGB simulation" workload. The video path can also be provided through `LIFEVIZ_SMOKE_VIDEO`. These file-backed smokes now fail if the file source does not publish enough fresh frame tokens, which catches freezes in the non-exact file-video handoff path automatically.
- `--smoke-test profile-current-scene` and `profile-current-scene-visible` load the persisted user config instead of a synthetic smoke scene, which is the correct profiling path once a performance problem only reproduces on the real saved stack. The visible variant keeps the normal desktop-composed window path alive so frame-pacing failures can be measured without manually reading the overlay.
- `--smoke-test profile-current-scene-fullscreen` extends that same real-scene path into fullscreen and now also validates the presented layout bounds, so the old "tiny centered/top-centered render rectangle" fullscreen failure is caught automatically instead of only by eye.
- `--smoke-test profile-current-scene-<144|240|480|720|1080|1440|2160>`, the visible variant, and the fullscreen variant force the real saved scene to a specific preset height before profiling, so a 720p-only or 1440p-only regression can be measured directly in the same presentation mode where it occurs.
- `--smoke-test profile-current-scene-presets`, `profile-current-scene-visible-presets`, and `profile-current-scene-fullscreen-presets` now sweep the full preset ladder from `144p` through `2160p`, which is the right baseline for tracking whether the app can realistically climb toward `4K`. The preset suite uses a shorter dwell per resolution than the single-target profiles so the whole ladder can run as a practical regression check.
- `--smoke-test profile-current-scene-interaction` extends that visible real-scene path by opening the Scene Editor, verifying that the main window does not remain interaction-throttled while the editor is open, then cycling the root context menu and comparing pre/editor/post frame-gap metrics so interaction-specific pacing regressions are caught automatically instead of being reproduced manually.
- `--smoke-test pacing-current-scene-visible-presets` and `pacing-current-scene-fullscreen-presets` turn pacing into a real pass/fail gate instead of a profile you have to interpret manually. They pin the real saved scene to a stable `60 fps` budget, sweep the realtime preset ladder (`144/240/480`), and fail if any run violates the configured frame-gap budget (`avg`, `p95`, `p99`) or underrun ratios (`frame_gap_over_25ms`, `frame_gap_over_33ms`, `frame_gap_over_50ms`).
- `--smoke-test pacing-current-scene-interaction` applies those same hard pacing checks to the `480p` Scene Editor/context-menu interaction case, and `pacing-current-scene-suite` bundles the visible preset ladder plus that interaction case into one regression pass.
- `--smoke-test pacing-current-scene-overlay-fullscreen-144` applies the same hard pacing gate with `Show FPS` enabled in fullscreen at `144p`, so the WPF timing/audio diagnostics are tested as part of the render budget instead of only by eye.
- `--smoke-test frame-pump-thread-safety` opens the real Scene Editor and then evaluates frame-pump interaction state from a worker thread. That catches cross-thread WPF access regressions in the timer-driven scheduler, including the `Window.IsActive` crash introduced by the custom-chrome frame-pump rewrite.
- GPU simulation-layer presentation now prefers shared D3D11 sim textures for both `Naive Grayscale` and `RGB Channel Bins`. CPU color-buffer readback is kept only for fallback, recording, or smoke-test verification paths.
- Audio analysis is no longer unconditional. The audio thread now skips waveform-history capture, 30-second debug-trace retention, and FFT/spectrum extraction unless the FPS/debug overlay is visible or a live feature requires spectral data (`Audio Granular` source animation or a simulation layer with reactive mappings that consume spectrum/band inputs). When those features are inactive, the app still computes the cheap envelope/beat signals needed for `Level -> FPS`, `Level -> Life Opacity`, and reactive seeding, but it avoids the heavier debug/spectrum work.
- When spectral data is needed, FFT/spectrum extraction is decimated to roughly `60 Hz` instead of running on every audio callback. The detector holds the last spectrum result between updates, which keeps `Audio Granular` and per-layer band/frequency mappings live while reducing audio-thread CPU cost and jitter.
- Live profiling now records finer-grained source capture timings too: `capture_window_frame_ms`, `capture_webcam_frame_ms`, `capture_sequence_frame_ms`, `capture_file_frame_ms`, and `capture_group_composite_ms`, in addition to the broader `capture_sources_ms` and `inject_capture_ms` buckets. That makes it possible to identify whether "inject/capture is slow" is really decoder acquisition, nested group composition, or later injection work.
- Live profiling now also records delivered file-video freshness directly: `capture_file_fresh_frame_ratio` and `capture_file_frame_age_ms` (plus sequence equivalents). That makes it possible to tell whether a "smooth sim, slideshow video" report is a stale-video problem or a render-pacing problem.
- Live profiling now also records per-source file-video freshness/age metrics using the source filename as a suffix. That matters when one file source freezes while another keeps updating, because aggregate freshness can still look acceptable in that mixed case.
- Current-scene profile smokes now wait through a warmup period and then validate the settled run against those freshness metrics. That change matters because startup/reconfigure noise can otherwise make a scene look healthy in smoke while it is still slideshowing in steady use.
- `--diagnostic-test` complements those smokes by running the normal saved-scene startup path instead of `App.IsSmokeTestMode`. Use it when smoke startup looks healthy but a real launch still diverges. The current diagnostic targets cover visible preset runs, fullscreen preset runs, interaction profiling, and a longer `720p` soak, and they write reports into the active build output's `profiles\` folder.
- Settled smoke/diagnostic validation now also checks `presentation_draw_fps`, not just frame-loop gap metrics. That closes the old gap where the app could report a healthy loop cadence while the actual presented surface was visibly stale.
- File-video sessions now start their decode worker on the first real capture request instead of immediately after probe. That ensures the worker starts once with the actual requested process mode (exact target vs adaptive/native) instead of launching a native-path worker and then immediately restarting it into a different mode.
- Long-lived file-video raw decode, sequence-advance, sequence-probe, and source-audio decode workers now run on dedicated long-running background threads instead of the shared threadpool. Those jobs are blocking media loops, so giving them dedicated threads avoids starving unrelated capture/render work during long sessions.
- Those long-running media workers and the spawned `ffmpeg` child processes now also run below normal by default. In **Low Contention Mode**, LifeViz goes further: it stops requesting the global 1 ms multimedia timer, keeps the frame pump on a lower-priority thread, and pushes decode processes toward idle/background scheduling. That reduces how hard LifeViz competes with other realtime apps and controller/audio stacks when multiple video/audio sources are active.
- The root context menu now exposes that behavior as a runtime tuning control. **Low Contention Mode** lowers LifeViz's process/frame-pump priority as well as media-worker pressure, **Decoder Threads** lets you cap `ffmpeg` decode fanout directly, and **Video Decode FPS** can force file/video sources to emit only `30` or `15` fps. Those settings are persisted and are applied to existing file/video/sequence sessions immediately by restarting their decode workers with the new limits.
- `--smoke-test dimensions` applies a real simulation height/depth change through `MainWindow`, forces the reference simulation layer into `RGB Channel Bins`, then drives the Scene Editor height dropdown in both Live Mode and deferred Apply mode, and fails unless every simulation layer plus the presentation surface height update together.
- `--smoke-test source-reset` clears the source stack, preserves passthrough state, re-adds a synthetic source, and fails unless the source composite is visible again after the stack comes back.
- `--smoke-test shutdown` opens and closes the real window, then fails if `MainWindow` captured any teardown exception during close.
- Simulation-layer render settings are now owned per layer: each layer carries its own life mode, binning mode, injection noise, life opacity, RGB hue offset, RGB hue speed, threshold window, and a list of reactive mappings. A separate global life opacity remains as a master multiplier across all simulation layers.
- Height selection is preset-driven (`144`, `240`, `480`, `720`, `1080`, `1440`, `2160`) in both the root context menu and the Scene Editor, so grid resize uses the same row-count choices everywhere.
- `--smoke-test shutdown` now opens the Scene Editor before closing the real main window, so owner/owned-window teardown regressions are caught too.
- CPU fallback pieces are now limited to source/presentation seams. Source compositing still has a CPU fallback path, but simulation itself no longer keeps a second CPU backend alive.

## Simulation Loop

- A coalesced one-shot background timer now drives the frame loop in `MainWindow.xaml.cs`; simulation stepping still starts from the selected framerate (15 / 30 / 60 / 144 fps), optional FPS oscillation, and optional global audio-reactive FPS boost, but enabled simulation layers now keep their own effective target FPS and their own step accumulators.
- A dedicated high-resolution frame-pump thread now owns cadence. Instead of relying on `System.Threading.Timer`, the scheduler thread tracks the next target presentation time with `Stopwatch`, uses a coarse sleep plus fine spin for the last sub-millisecond slice, and posts at most one pending frame callback to the dispatcher at a time. The dispatcher callback normally runs at `Render` priority, and the cadence state is explicitly reset after native move/resize throttling or a simulation-dimension change. That substantially reduces baseline underruns and timer-queue jitter without going back to `CompositionTarget.Rendering`. The actual frame work still runs on the WPF UI thread, and LifeViz now reserves interaction throttling for true native move/resize loops instead of automatically dropping to a low-rate/background-priority path just because the root context menu or Scene Editor is open.
- Any interaction state that the background timer needs is now cached into thread-safe fields on the UI thread first. The timer thread no longer reads WPF-owned properties such as `Window.IsActive` directly, which avoids cross-thread `InvalidOperationException` crashes.
- Source capture/composite refresh is now decoupled from simulation stepping. `MainWindow` refreshes the source stack every render tick, then only injects into simulation layers on frames where a layer actually needs a step. That keeps passthrough and file/video underlay smooth even when no simulation layers are enabled or when a layer's effective sim FPS is lower than render FPS.
- Shared GPU sim/underlay presentation now performs one synchronized producer flush at the final handoff into `PresentSimulationComposition(...)` whenever shared producer resources are actually in play. That avoids the stale-shared-texture case where the sim appears to update at ~1 FPS visually even though the frame loop and simulation-step counters are still advancing.
- The main output window now removes the native non-client title bar entirely (`WindowStyle=None`, `ResizeMode=NoResize`) and replaces it with custom chrome plus manual pointer-driven move/resize zones. That means ordinary window dragging/resizing no longer enters the OS move loop, which is necessary if render cadence is going to be meaningfully decoupled from shell interaction. Native fullscreen still uses the separate fullscreen toggle path.
- Window movement is no longer limited to the chrome strip. The render background itself now participates in the same pointer-driven drag path, and double-clicking that background toggles fullscreen, so the output surface behaves like a projector-first canvas rather than a conventional titled window.
- Manual resize now owns aspect enforcement during the drag. `MainWindow_OnSizeChanged` no longer fights the custom resize path, so releasing a resize grip no longer snaps the window back to an old size.
- Entering or exiting fullscreen now explicitly resets frame-pump cadence as part of the mode switch, which prevents stale scheduler timestamps from poisoning the first steady-state fullscreen frames.
- Applying a new simulation height/depth now also clears the last composite cache, forces a surface rebuild, and issues an immediate redraw so a dimension change cannot remain visually stuck on an old frame until later activity arrives.
- Each simulation step (unless paused) captures every active source, composites them (per-source blend modes, ordered by the stack, including layer groups), injects that final composite into each enabled simulation layer's current generation (using each layer's Direct/Inverse input function), applies optional audio seeding (continuous level-based and/or beat-triggered), advances enabled layers once, then calls `RenderFrame()`. Injection no longer pushes an extra history slice before stepping, which keeps autonomous patterns crisp instead of duplicating them in the depth trail.
- When target simulation FPS exceeds the effective UI tick cadence, LifeViz runs multiple simulation steps per timer tick (bounded catch-up) so high-rate modulation remains visible in evolution speed.
- The engine maintains a depth stack (`List<bool[,]>`) where index 0 is the newest frame.
- Clearing all sources no longer forces passthrough off. An empty source stack still renders black when there is no underlay to show, but the passthrough preference is preserved so re-adding sources restores underlay display immediately.
- On the GPU presentation path, the finished simulation frame is uploaded into a dedicated D3D11 texture, the passthrough underlay is uploaded separately when needed, and a fullscreen shader pass resolves the selected composite blend mode into the `DrawingSurface` color target. Source compositing can now also happen in D3D11, and the grayscale-GPU injection path can consume that composite directly without rebuilding CPU masks first. On the normal live GPU path, passthrough underlay now uses the shared GPU composite surface directly instead of forcing a CPU composite readback just for final presentation; CPU composite readback is still required for recording and for any simulation layer that falls back to CPU mask injection on that frame.
- The visible render host no longer wraps the presentation surface in a WPF `Viewbox`. The GPU `DrawingSurface` and fallback `Image` now scale with `Stretch=Uniform` inside a plain black host, which removes a large-window/fullscreen WPF scaling layer that could make video presentation look choppy even when simulation timing stayed healthy.
- `DrawingSurface` has no natural bitmap aspect of its own, so the GPU presentation backend now computes the displayed DIP rect explicitly from the sim-surface aspect and the current host size. That keeps fullscreen presentation filling the expected letterboxed/pillarboxed rect instead of collapsing to a square when WPF has only the host bounds to work from.
- The presentation pass no longer forces an explicit D3D context flush every frame. WPF interaction latency during window drag/menu use is highly sensitive to UI-thread GPU sync points, so presentation now queues work normally and lets the compositor pace it.
- Per-layer simulation backend ownership is now decoupled from `MainWindow`; removing/reordering layers disposes any orphaned simulation backend instances, which is necessary once layers start owning GPU resources instead of pure managed CPU state.
- `GpuSimulationBackend` now covers both life modes. In `Naive Grayscale`, the backend can inject in two ways: CPU-built masks upload into a D3D11 mask texture for fallback compatibility, or the preferred path samples the final GPU composite texture directly inside a compute shader using the layer's threshold window, direct/inverse input function, injection mode, PWM period, and injection noise. In `RGB Channel Bins`, the backend keeps the combined depth stack on GPU, injects either uploaded RGB mask textures or the final GPU composite directly, applies inverse hue-rotation mapping before channel gating, steps each channel slice independently on GPU, and renders the final RGB color texture there too.
- `InjectCaptureFrames()` keeps the common path GPU-only, but if a layer needs CPU mask fallback on a frame where the composite was built without CPU readback, it now lazily rebuilds the composite once with CPU readback enabled and uses that for fallback injection. That preserves correctness without forcing every frame through CPU readback.
- On the normal GPU presentation path, grayscale GPU simulation layers now share their rendered color textures with `GpuPresentationBackend` through D3D11 shared-resource handles. Final simulation-layer compositing now also owns per-layer RGB hue rotation on the GPU: the CPU computes each layer's current hue `sin/cos` once, then the fullscreen final-composite shader applies that hue to the sampled layer texture before blend. CPU hue rotation is only retained for explicit CPU fallback cases such as recording.
- Shared GPU simulation textures are now submitted once per frame, not once per layer. `GpuSimulationBackend.TryGetSharedColorTexture()` no longer flushes the shared D3D11 context on every layer query; instead the main render path flushes the shared device once only when at least one shared simulation texture will actually be presented. That removes redundant UI-thread GPU sync points in multi-layer RGB scenes.
- The GPU simulation backend now has a runtime smoke test path. It validates D3D11 device/shader initialization, grayscale injection/step/render, RGB injection/step/render, binary/fill color-buffer output, and mode switching across both life modes.

## Color Encoding

- The depth is partitioned into three slices (R/G/B). If depth = 24, each slice gets 8 layers; remainders are distributed so `Depth % 3` spreads extra layers across R and G first.
- For each slice, the engine walks the relevant frames and treats alive cells as set bits, forming a binary number. That integer is normalized against the maximum for that bit-length to derive a 0-255 channel value.
- The grayscale GPU simulation backend reproduces that same slice math in its render compute shader, including both `Fill` and `Binary` binning, so grayscale output parity does not depend on per-cell CPU reads.
- In *RGB Channel Bins* mode, hue is resolved per simulation layer, not globally. Each layer's final hue is:
  - base `Hue Offset`
  - plus time-based `Hue Speed`
  - plus any reactive `Hue Shift` mappings (for example `Frequency -> Hue Shift`)
- Dominant frequency uses the same logarithmic note-scale normalization as the magenta debug trace, so equal musical intervals produce roughly even hue travel.
- RGB capture injection uses the inverse of that same per-layer hue rotation when building per-channel masks, remapping source colors into the rotated bin basis before mode-specific gating. This keeps channel activity aligned to the currently shifted palette.
- The renderer writes BGRA bytes into `_pixelBuffer` by compositing simulation layers in the configured simulation-layer order (editable in Scene Editor), each with its own blend mode; default stack is Positive (Direct/Additive) first, then Negative (Inverse/Subtractive) on top, then presents that buffer through the active render backend.
- For simulation-layer `Subtractive` blending, subtraction is performed against the already composited simulation pixel. With passthrough off, subtractive-only stacks use white as the baseline (subtractive identity). Additive+subtractive-only mixed stacks use a neutral gray baseline (50% normalization bias) so both layers stay legible when combined; other mixed blend families use black baseline. With passthrough underlay precomposited in-buffer, additive/subtractive-only stacks are resolved in a signed delta path, and mixed add+sub pixels apply underlay-aware headroom normalization before final clamp to reduce order bias and prevent cross-family washout.

## Source Capture Injection

1. `WindowCaptureService` enumerates all visible, non-minimized windows (excluding LifeViz itself), grabs their DWM extended frame bounds to get physical pixel sizes (avoids DPI virtualization cropping), and captures each active window via BitBlt into a `System.Drawing.Bitmap`. `WebcamCaptureService` uses the modern WinRT `MediaCapture` API, which provides a high-performance path to stream frames directly into reusable memory buffers, avoiding the GC pressure associated with frequent allocations during capture. `FileCaptureService` loads static images (including WEBP) and GIFs via WPF bitmap decoders and streams video frames via `ffmpeg`; animated files loop, video sequences advance on end, and frames are converted to BGRA buffers for compositing.
2. File-video sessions now keep a short queue of decoded raw frames plus a small reusable buffer pool instead of a single "latest frame" slot. Brief UI/render stalls therefore consume buffered decoded frames on the next ticks instead of immediately overwriting unread video frames and presenting visible playback skips. The decoder no longer emits periodic per-frame heartbeat logs during normal playback.
3. Scene Editor simulation-layer dropdowns bind to `LayerEditorOption` values, and `LayerEditorOption.ToString()` now returns the user-facing label. That keeps the selected value readable inside the custom dark ComboBox template instead of showing the CLR type name.
4. Each capture is downscaled to the grid size and stored as BGRA using the per-source fit mode (Fill default, plus Fit/Stretch/Center/Tile/Span); source downscale now uses small supersampled filtered sampling (2x2 taps over each destination pixel footprint) to reduce vertical/horizontal banding and aliasing in live imagery before injection, while buffers are still reused per window/webcam to avoid per-frame allocations. File-video downscale also avoids `Parallel.For` for small outputs, because low-resolution scenes spend more time in threadpool overhead than in the actual sampling work.
5. File-video downscale now short-circuits the common adapted-video cases before falling back to the supersampled sampler:
   - exact-size frames are copied directly
   - equal-width fill/center frames use centered vertical crop
   - equal-height fill/center frames use centered horizontal crop
   - equal-width/equal-height fit/center frames use centered pad copy
   These fast paths target low-resolution adapted decode outputs where the old supersampled per-pixel sampler was needlessly expensive and could leave underlay video visibly stale even while simulation/render timing looked healthy.
6. When a video layer does not need native source pixels and its fit mode is `Fill`, `Fit`, or `Stretch`, `VideoSession` now asks `ffmpeg` to emit exact target-sized BGRA frames directly only when that target still fits inside the native source bounds:
   - `Fill` uses `scale ... force_original_aspect_ratio=increase` plus `crop`
   - `Fit` uses `scale ... force_original_aspect_ratio=decrease` plus `pad`
   - `Stretch` uses direct `scale`
   In those modes, the decoded frame can bypass the CPU downscale worker entirely and publish straight to the compositor/injection path. If the requested preset would force decoder-side upscaling beyond native size, the session stays on adaptive/native decode and lets the GPU compositor do the upscale instead.
7. Even when a video session is not using exact target output, it now publishes its current decoder-sized frame directly into the compositor path instead of forcing a CPU resample to engine size first. That means adaptive/native video decode sizes can still stay fresh while the GPU compositor handles the final scale to the grid. The published frame also carries its true buffer dimensions through the capture path, so the compositor does not mistake a native/adaptive frame for an already-target-sized one. `CaptureFrame()` now returns those decoder-sized buffers as-is instead of discarding them for not matching the requested target size, which fixes the source-specific freeze where smaller videos would stop updating as soon as they left the exact-target decode path.
8. On `GpuRenderBackend`, sources are uploaded as D3D11 textures and composited in stack order on a GPU render target using the same fit modes, mirror handling, animation transforms, keying, opacity, and blend modes (Normal, Additive, Multiply, Screen, Overlay, Lighten, Darken, Subtractive). File/video-sequence sources can hand the compositor their decoder-sized frame directly, and the GPU does the final fit/scale work instead of the old CPU video resampler. The compositor uses a ping-pong render-target pair and can export the final composite as a shared D3D11 texture for direct grayscale-GPU simulation injection. On failure or unsupported initialization, it falls back automatically to `CpuSourceCompositor`. Layer groups still recurse through `BuildCompositeFrame`, so grouped composites benefit from the same GPU path when the backend is active.
9. File-video sessions now also adapt the `ffmpeg` decode/output size to the current simulation target whenever native source pixels are not required. Instead of always decoding raw BGRA at native-ish resolution and then downscaling inside LifeViz, low-resolution scenes restart the file-video pipeline at a process size that preserves source aspect while covering the current grid. That cuts raw frame bandwidth and internal downscale pressure substantially on multi-video `144p`/`240p` scenes, which is where the "smooth life, slideshow underlay video" failure mode showed up.
10. File-video restarts now preserve the last published frame while the new `ffmpeg` worker spins up. Without that, a size change like `480p -> 1080p` could momentarily clear `_readyDownscaled`/`_readyRaw` and present black until the replacement decode stream produced its first frame.
11. Video-sequence layers keep presenting their last good frame while the next clip session is opened asynchronously. Sequence handoff is still a cold clip open today, but the compositor no longer drops to black just because the next clip has not produced its first decoded frame yet. Sequence sessions now also cache clip probe metadata ahead of time, so an advance no longer has to pay the full synchronous duration/dimension probe cost during the live handoff.
12. The GPU final-composite path no longer forces redundant producer-device flushes before every present. Shared underlay/simulation textures are still consumed through the GPU final-composite shader, but the hot path no longer synchronizes the producer device twice per frame, which was the main cause of the render-side `720p` stalls after source capture was fixed.
13. The composited downscaled buffer feeds the injection path for every enabled simulation layer. Each layer chooses its own input function (Direct `y=x` or Inverse `y=1-rgb`), injection mode (Threshold/Random Pulse/Pulse Width Modulation), and threshold window (min/max + invert) before mode-specific gating. Life mode and binning mode remain global, and noise remains global. In **Threshold** mode, each layer's threshold window acts as a boolean gate. In **Random Pulse** and **Pulse Width Modulation**, that same window shapes injection intensity before probability/duty-cycle evaluation (including inverted windows), so threshold sliders influence all injection modes. If a source disappears, it is removed automatically; removing the last source restores the default aspect ratio (and webcam capture is released).
14. On pure binary inputs, injection modes can legitimately converge. For example, with a threshold window of `0.5..1.0`, black maps to `0` drive and white maps to `1` drive, so Threshold, Random Pulse, and PWM all inject identically on those pixels. To verify that mode switching still works, use a mid-gray source or the `gpu-injection-mode` smoke test rather than a pure black/white pattern.

## Passthrough Underlay

- When **Passthrough Underlay** is enabled, the composited buffer is preserved for presentation as well as injection.
- Primary path: when the source composite matches the simulation grid, rendering copies the underlay into `_pixelBuffer` first, then blends simulation layers on top in configured order using each layer's own blend mode. This keeps additive and subtractive simulation behavior consistent while passthrough is on.
- In that primary path, simulation is applied as a delta from the simulation baseline onto the underlay pixel. This keeps life contrast comparable with passthrough off instead of flattening/attenuating both layers.
- The same signed additive/subtractive passthrough model now applies when the underlay stays on the shared-GPU presentation path instead of being precomposited into the CPU pixel buffer. That matters because per-layer reactive opacity can otherwise make subtractive identities behave like real subtraction and visually fade the passthrough underlay after a source-stack change flips the app onto the GPU-underlay path.
- For additive/subtractive-only stacks, passthrough uses per-family composition over the underlay with headroom normalization. Contributions are measured relative to each blend mode's identity first: additive uses raw layer color (`0` = no effect), while subtractive uses inverse distance from white (`255 - layerColor`, so white = no effect). Additive contribution is then scaled by underlay darkness (`1 - underlay`), and subtractive contribution is scaled by underlay brightness (`underlay`), which prevents one family's steady-state bias from clipping out the other when both are enabled.
- Fallback path: if the GPU composite shader/resources are unavailable for any reason, the backend falls back to CPU pre-blending of the underlay before upload so passthrough parity is preserved without depending on WPF shader effects.
- `Life Opacity` is applied to simulation contribution only; passthrough underlay intensity is not attenuated by life opacity.
- When reactive opacity drives every simulation layer to zero on a frame, the GPU path now still executes the underlay-only final composite instead of dropping back to a black simulation frame plus overlay fallback. That keeps passthrough visually stable during silence or low-level sections.
- When there are zero active simulation layers, `MainWindow` now still refreshes the source composite through the normal frame loop and then presents the CPU-readable passthrough composite directly as the final frame. That removes the zero-layer edge case where passthrough could stay black simply because source/composite refresh had been tied to simulation stepping.
- Underlay rendering is skipped when no sources are active.

## Layer Animations

- Each source (including layer groups) can stack animations: Zoom In, Translate, Rotate, Beat Shake, Audio Granular, Fade, and DVD Bounce (with an adjustable scale). Every animation includes a cycle length in beats, so long fades can stay locked to BPM; Beat Shake uses audio beat timestamps when *Sync to Audio BPM* is enabled (falling back to the animation BPM for a timed pulse when no audio device is configured), exposes an intensity control, and ignores speed/cycle settings so its amplitude stays consistent per beat. Audio Granular now uses the fast peak-followed level signal with a soft gate, nonlinear compression, and bounded transform caps so it rides hits and releases quickly without hanging on a long average; it also applies per-animation 3-band EQ gains (Low/Mid/High) to shape zoom/translation/rotation response and supports higher intensity scaling up to 1000%. Neutral EQ (100/100/100) now maps to a stronger baseline response so the effect is clearly visible without pushing intensity to extremes.
- Video sources are decoded with `ffmpeg` into raw BGRA frames. If a video fails decode (blank frames or zero reported dimensions), LifeViz can auto-transcode to H.264 using `ffmpeg` (cached under `%LOCALAPPDATA%\\lifeviz\\video-cache`). While transcoding, the layer is skipped so it doesn't black out the stack. If `ffmpeg` is missing or the transcode fails, a warning is shown so you can transcode manually.
- Video/YouTube layers can optionally play source audio (default off) through an in-app audio pipeline (`ffmpeg` PCM decode + NAudio output). YouTube playback resolves video and audio URLs independently so audio can still play when the render path uses a video-only stream. If a source has no playable audio stream, the layer logs a warning and remains silent. Audio startup is debounced to prevent duplicate launches during rapid state changes, the decode input is clocked in real-time to avoid fast decode/restart churn, and the pipeline is torn down when audio is toggled off, the source is removed, or the app shuts down. When audio is enabled mid-playback, the decoder seeks to the current estimated video time so audio starts near the active frame instead of restarting from the beginning. Source audio gain is applied as `master volume * per-layer volume`, with a global master toggle/slider plus per-layer volume controls (including video sequences), and both are persisted/restored with source settings.
- Video transport controls support pause/play and scrubbing. Seeking restarts decode at the requested offset (best effort) and keeps pause state; pausing stops both video decode and source audio, resuming restarts decode from the paused position. Single video/YouTube sources run in loop mode continuously, while video sequences loop at the list level.
- Animations are evaluated during compositing by transforming the destination sampling coordinates before the fit-mode mapping, so the animation affects both injection and rendering.
- All animations share a global BPM (default 140) with per-animation half/normal/double time and forward or reverse (ping-pong) loops.

## Audio Reactivity

- Audio input is analyzed continuously (`AudioBeatDetector`) for energy and beat events.
- Device selection supports both capture inputs (microphones, via AudioGraph) and render outputs (via WASAPI loopback, including a default system output entry).
- `Input Gain` scales raw samples before energy, beat, and FFT analysis. The UI range is now tuned for real use (`0.00x..2.00x` in `0.05x` steps) instead of a huge rescue/debug range, and gain presets are tracked separately for input-device and output-device capture paths.
- Spectrum outputs are signal-gated: when RMS falls below a small floor, reported bass/frequency collapse to zero to avoid noisy "fake frequency" readings on silence. In addition to overall dominant frequency, the detector now also tracks dominant frequencies in bass, mid, and high bands for the debug overlay.
- Loudness normalization is tuned for typical program material rather than near-full-scale samples, so normal loopback playback reaches a much larger portion of the 0-1 range before clamping.
- The displayed `Audio: %` meter uses a rectified short-window max envelope signal with more normalization headroom than the slower loudness path. That keeps it responsive to hits while fitting the absolute waveform much more closely than averaged or peak-held signals.
- When **Show FPS** is enabled, `MainWindow` also draws two lightweight WPF overlay panels: a timing panel with an orange polyline from rolling frame-gap history and a red dashed frame-budget line, and an audio panel with a white waveform range trace from rolling min/max waveform buckets captured in `AudioBeatDetector`, a yellow polyline from a rolling rectified max-envelope history, a cyan polyline from rolling bass-energy history, and four note-scale frequency traces: magenta for overall dominant frequency, blue for bass-band dominant frequency, green for mid-band dominant frequency, and red for high-band dominant frequency. The text overlay also shows a recent rolling-average `Work:` breakdown for `frame_total_ms`, `build_composite_ms`, `inject_capture_ms`, `simulation_total_ms`, and `render_call_ms`, so visible-window pacing failures can be separated from actual frame-work cost. All traces use the same fixed 30-second window, with new data entering on the right and scrolling left. Yellow and cyan share the waveform zero baseline, the timing traces use a full-height `0..50 ms` scale, and the frequency traces use the full panel height on a logarithmic note-like scale. To keep WPF overlay cost bounded, the graph is resampled to a capped sample count and refreshed on split cadences: the timing panel repaints faster than the audio panel instead of forcing both to redraw together every time. The audio traces now come directly from `AudioBeatDetector`'s rolling history buffers, so the overlay is no longer re-sampling bass/frequency histories through additional UI-owned queues. The white trace uses bucket min/max geometry rather than one sample per bucket, which preserves waveform shape at long time scales. This stays outside the simulation render path so it does not affect GPU compositing.
- The FPS text now distinguishes `Present`, `Loop`, and `Sim`. `Present` is derived from actual render-backend draw count, `Loop` is the internal frame callback cadence, and `Sim` is simulation steps per second. That makes visible stale-present bugs diagnosable without conflating them with the scheduler cadence.
- The overlay cost itself is guarded by `pacing-current-scene-overlay-fullscreen-144`, which turns on `Show FPS` for the real saved scene in fullscreen and fails if the overlay pushes frame-gap pacing outside the same hard underrun budget used by the non-overlay pacing suites.
- Startup is guarded by a recovery flag under `%APPDATA%\\lifeviz`. If the previous session dies before `MainWindow` finishes loading, the next launch forces a safe config subset (`<=480p`, `<=60 fps`, windowed, overlay off, `Level -> Framerate` off) before initializing the saved scene so a bad persisted fullscreen/fps combination cannot brick the app on the next run.
- Legacy whole-scene `Level -> Framerate` and `Level -> Life Opacity` controls are no longer the canonical modulation path. On config load, those legacy flags are migrated into equivalent per-layer `Level -> Framerate` and `Level -> Opacity` mappings (`amount = 1 - floor`) and then cleared, so the runtime does not keep two competing systems alive. The older per-layer `AudioFrequencyHueShiftDegrees` field is also load-only now: it can still migrate old configs/projects into a `Frequency -> Hue Shift` mapping, but editor applies and later saves only persist the canonical reactive mapping list so cleared mappings do not reappear on restart.
- *Level -> Seeder* converts the same loudness signal into per-step seed burst counts (with configurable max bursts), injecting selected patterns continuously as level rises.
- When per-layer `Level -> Framerate` mappings drive a simulation layer near zero, global level seeding still injects bursts so reactivity remains visible in the frame output.
- *Beat -> Seeder* watches beat count edges and injects configurable seed bursts (Glider, R-pentomino, Random Burst) with a configurable cooldown to avoid runaway injection.
- Audio-reactive controls are gated by audio device selection (`Audio Source`); with no device selected, reactivity remains configured but inactive.
- Simulation layers can now own modular reactive mappings. Each mapping resolves one normalized input (`Level`, `Low`, `Mid`, `High`, overall `Frequency`, or band-limited frequency) into one output (`Opacity`, `Framerate`, `Hue Shift`, `Hue Speed`, `Injection Noise`, `Threshold Min`, or `Threshold Max`) plus an amount.
- Per-layer mappings are keyed to audio device selection rather than the legacy global audio-reactivity master toggle. That keeps the new modular system independent from the old whole-scene controls while still falling back to a neutral state when no device is selected.
- `Opacity` and `Framerate` mappings use multiplicative `0..1` strength against the layer's base opacity / the current global simulation target FPS. `Hue Shift` mappings add `0..360` degrees on top of the layer's base hue offset and hue-speed animation. `Hue Speed` adds up to `180 deg/s` on top of the layer's configured hue-speed animation. `Injection Noise` adds directly to the layer noise amount, `Threshold Min` raises the effective lower bound, and `Threshold Max` lowers the effective upper bound; the runtime re-normalizes the min/max pair if they cross.
- Band-energy inputs are normalized logarithmically from FFT magnitudes inside their own bins. Band-frequency inputs are log-normalized within their own frequency bands (`20..250`, `250..2000`, `2000..8000 Hz`) so the mapping behaves more like musical pitch space than linear Hz.

## Recording

- **Start Recording** captures the final output buffer, then applies a pixel-perfect integer upscale to the nearest HD height (720/1080/1440/2160 when divisible). **Recording Quality** chooses between lossless FFV1 in MKV (compressed, requires `ffmpeg`), crisp H.264 in MP4 (Windows Media Player compatible, requires `ffmpeg`), uncompressed RGB (AVI), or H.264 tiers (High/Balanced/Compact). H.264 uses quality-based VBR with bitrate caps to preserve sharp pixel edges without runaway file sizes.
- Recording uses the same final composite logic as the renderer (including passthrough precomposition/fallback behavior and invert composite), then writes frames at the configured framerate.

## Life Modes

- **Naive Grayscale** (default): each enabled simulation layer uses a single simulation derived from a thresholded luminance mask of the capture, with per-layer Direct/Inverse input function and per-layer threshold window controls.
- **RGB Channel Bins:** each enabled simulation layer uses three independent simulations in R/G/B bins. Captures inject per-channel masks into each layer with per-layer Direct/Inverse input function; propagation stays within each channel slice (e.g., depth 24 yields bins 0-7, 8-15, 16-23).
  - Optional `RGB Hue Shift` rotates both output colors and input channel mapping (via inverse transform during mask generation), so the simulation's per-channel behavior changes with hue angle.

## Binning Modes

- **Fill** (default) - channel intensity is the ratio of alive cells within the bin (frames in that channel slice). Produces smoother intensity ramps across depth.
- **Binary** - original bit-packed depth encoding; most significant bits represent newer frames, normalized to 0-255.

## Z-Stack Effect

Because every new frame pushes down the history stack, movement leaves chromatic trails. Injected window captures therefore colorize based on how recent pixels were alive, giving a time-scrubbed representation of the source window.

## Performance Notes

- Columns = round(rows * aspectRatio). The primary source replaces the default 16:9 ratio with its current ratio; removing all sources restores 16:9, and the selected height (rows) stays fixed while the width adapts.
- Capture buffers are pooled (including the raw window/webcam readback), and only the downscaled composite is kept alive, removing GC spikes that caused occasional lurches.
- Random fill uses 35% seed density to encourage interesting evolution when no source is selected.
- Simulation compositing is no longer CPU-bound on the normal path. Passthrough and final presentation prefer a D3D11-backed WPF surface (`DrawingSurface`) with fullscreen shader passes for underlay blending and final simulation-layer composition, and GPU sim layers now feed that pass through shared D3D11 textures instead of CPU readback. CPU byte-buffer blending remains only as a presentation fallback or recording path; simulation layers themselves are no longer CPU-owned. CPU source readback is still required for passthrough/recording cases that need host-visible pixels. Source capture/downscale uses filtered supersampling, while the CPU compositor uses fast direct copies for already-aligned layers and bilinear remapping only when a source needs transformed or fit-mode resampling. Final display scaling remains nearest-neighbor so cell edges stay crisp. Window capture uses GDI BitBlt, so extremely large or numerous sources may require smaller grids or lower tick rates if compositing falls behind.
- WPF input interaction should stay off the render surface itself. Hover/mouse hit-testing against the GPU `DrawingSurface` can introduce avoidable UI-thread overhead, so the presentation host is treated as display-only and does not participate in focus or hit testing.
